\subsection{Service, Infrastructure \& Resources}

Our considered system adopts a container-based architecture composed of three main elements:

\begin{itemize}
    \item \emph{Services}: Logical groupings of CNFs. Each service $s \in S$ is defined by a set of CNFs denoted as $F_s$.
    \item \emph{CNFs}: Individual containerized network functions.
    \item \emph{Nodes}: Physical infrastructure elements with finite computational resources.
\end{itemize}

We primarily focus on processing (CPU) and memory (RAM) as key resources. Each node $N$ is defined by its allocatable resources $\{CPU^{alloc}_n, RAM^{alloc}_n\}$. For each CNF $f \in F$, the resource requirements are given as $f = \{CPU^{req}_f, RAM^{req}_f\}$. Regarding CPU, we distinguish between allocated resources and the resources currently in use. Specifically, each CNF $f$ has a time-dependent CPU workload at time $t$, denoted by $W^{CPU}_f(t)$ and defined in~\eqref{eq:cpu_usage}.

\begin{align} \label{eq:cpu_usage}
    CPU_n^{\%}(t) = \frac{\sum_{f \in F_n} W_f^{CPU}(t)}{CPU_n^{alloc}}.
\end{align}

\subsection{Energy}

An energy consumption model is associated to each node $n$ and a renewable energy generation model is considered when available on the node.

\subsubsection{Energy consumption model}

We define the power consumption $P_n$ of a node $n$ as a function of its CPU utilization. In our work, the adopted model $PM_n$~\eqref{eq:power_model} is a polynomial of degree $m$ with coefficients $a_i$ obtained via polynomial regression from experimental power data on the platform Grid'5000~\cite{Grid5000} as displayed in Fig. \ref{fig:power_models}. 

\begin{align}\label{eq:power_model}
    PM_n(x) = a_0 + a_1 x + a_2 x^2 + \dots + a_m x^m.
\end{align}

To create a model correlating CPU usage with power consumption, we designed an experimental method. We executed a CPU-intensive script on a node, simultaneously measuring both CPU utilization and power consumption.

To vary the CPU load, we used \texttt{stress-ng}, a Linux-based stress-testing tool. This tool allows precise control over the number of CPU cores used, their load percentage, and the duration of each workload.

CPU usage was measured using the \texttt{psutil} Python library (\texttt{cpu\_percent} function), which provides real-time CPU utilization metrics. On Linux, \texttt{psutil} can use the \texttt{/proc} filesystem and make calls to \texttt{sysinfo}, \texttt{getloadavg}, etc.

Power consumption was retrieved using the Grid'5000 API. The platform uses wattmeters to measure node power draw at one-second intervals. All metrics were retrieved post-experiment using the experiment's timestamps.

Our script follows a three-phase approach:

\begin{enumerate}
    \item \emph{Idle phase}: 30 seconds without load.
    \item \emph{Fixed CPU load test}: Sequential stress tests at 10\% increments (10\%, 20\%, ..., 100\%) for one minute each, with a 5 seconds cool down, applied across all cores.
    \item \emph{Mixed workload test}: Randomized workloads with variable core usage, load percentages, and durations.
\end{enumerate}

\subsubsection{Energy generation model}

When renewable generation is available on a node $n$, we define the green power output as a time-varying function $P^{green}_n(t)$. The residual power drawn from the electrical grid (that we call brown power in our work) is then defined by~\eqref{eq:brown_power}.

\begin{align}\label{eq:brown_power}
    P^{brown}_n(t) = 
    \begin{cases}
        P_n(t) - P^{green}_n(t) & \text{if } P_n(t) > P^{green}_n(t), \\
        0 & \text{otherwise}.
    \end{cases}
\end{align}

In our model, renewable generation can either come from solar or wind sources.

\emph{Solar power generation} is modeled using a normal distribution-based irradiance profile, estimated via Kernel Density Estimation (KDE) as used in~\cite{trashchenkovKernelDensityEstimation2018}. The instantaneous solar output is then $P^{solar}_n(t) = \eta_n \, S_n \, I_n(t)$, where $\eta_n$ is the panel efficiency, $S_n$ the panel surface area, and $I_n(t)$ the irradiance at time $t$. The process is detailed in Fig.~\ref{fig:solar_generation}.

\begin{figure}[!h]
    \centering
    \includegraphics[width=\linewidth]{figures/chapter3/solar_generation.png}
    \caption{Step-by-step visualization of solar power generation from a normal distribution using KDE.}
    \label{fig:solar_generation}
\end{figure}

\emph{Wind power generation} model is derived from wind speed generation using a Weibull distribution based on the work in~\cite{megantoroModelingUncertaintiesActive2025a}. The final power output can be computed with~\eqref{eq:wind}, where $P_n^{rated}$ is the rated power output, $v_n(t)$ the wind speed, and $v_n^{cut\_in}$, $v_n^{rated}$, $v_n^{cut\_out}$ the cut-in, rated, and cut-out speeds, respectively. This is also explained by Fig.~\ref{fig:wind_generation}. For both renewable generation models, we add a Gaussian noise to account for natural variability.

\begin{align}\label{eq:wind}
    P^{wind}_n(t) = 
    \begin{cases}
        P_n^{rated} \cdot \dfrac{v_n(t) - v_n^{cut\_in}}{v_n^{rated} - v_n^{cut\_in}} & \text{if } v_n^{cut\_in} \leq v_n(t) < v_n^{rated} \\
        P_n^{rated} & \text{if } v_n^{rated} \leq v_n(t) \leq v_n^{cut\_out} \\
        0 & \text{otherwise.}
    \end{cases}
\end{align}

\begin{figure}[!h]
    \centering
    \includegraphics[width=\linewidth]{figures/chapter3/wind_generation.png}
    \caption{Step-by-step visualization of wind power generation from a Weibull distribution using KDE.}
    \label{fig:wind_generation}
\end{figure}

This stochastic approach enables us to evaluate our methodology across diverse power generation models, eliminating reliance on a fixed dataset. However, for practical applications, using real-world measurements may be more appropriate.

\subsection{Monitoring}

Monitoring ensures continuous visibility into the state and performance of both physical and virtual entities. It encompasses a wide range of components, from physical nodes to virtual services like our CNFs. Energy-related infrastructure, including power consumption and renewable energy generation, is also closely tracked.

The metrics collected are diverse and purpose-driven. Physical resources like CPU and RAM are monitored alongside virtual resource utilization. Energy metrics, including consumption and generation from renewable sources, are also monitored. Additionally, the system tracks the status and performance of nodes and services, capturing health indicators, uptime, and response times.

The data gathered serves multiple purposes. It provides real-time insights for orchestrators, enabling them to make informed, dynamic decisions. It also acts as a valuable dataset for training machine learning models, supporting predictive analytics and long-term optimization. Finally, this data is transmitted to analysis modules, where it is processed to uncover trends, anomalies, and opportunities for improvement.

\subsection{Analysis}

The analysis module transforms raw monitored data into usable intelligence. By aggregating and correlating metrics across the system, it identifies patterns, detects anomalies, and highlights lack of resources. This processing is not just reactive, it can also be used to make predictions, such as energy availability based on renewable generation trends or anticipating workload demands during peak periods.

These insights directly influence the orchestrator’s decisions. For example, predictions about energy shortages or workload spikes can trigger proactive adjustments, ensuring the system remains efficient, reliable, and aligned with operational goals.


\subsection{Orchestration}

Orchestration is the decision-making engine of the system, responsible for dynamically allocating resources and managing services in accordance with predefined policies and real-time insights. It ensures that services are placed optimally, balancing factors such as performance, energy efficiency, and cost. Resource allocation is not static, it evolves based on demand, availability, and policy constraints. A key function of orchestration is the dynamic migration of CNFs.

In our problem, we consider an Edge service orchestration problem where services, composed of multiple CNFs, must be deployed on a set of nodes with limited resources and subject to unpredictable node departures. The objective is to find optimal \textit{service placement} and \textit{service placement reconfiguration} strategies that makes the best use of available energy resources while ensuring service continuity.

\subsubsection{Service Placement}

Service placement orchestrator assigns each CNF to a physical node while satisfying resource constraints and optimizing multiple objectives. We define a binary decision variable $X(n,f,t) \in \{0,1\}$ that equals $1$ when a CNF $f$ is placed on a node $n$. Each CNF must be placed on exactly one node, and the total resource requirements of CNFs placed on a node must not exceed its allocatable resources.

\subsubsection{Service Placement Reconfiguration}

We then define service placement reconfiguration as the process of migrating CNFs from a node to another in response to infrastructure changes, such as energy variations or node departures. We use the same variable $X(n,f,t)$ and introduce a new migration variable $M(n,f,t) \in \{0,1\}$ that equals $1$ if CNF $f$ is migrated to node $n$ at time $t$. The reconfiguration process must also satisfy the same resource constraints.

\section{Optimization Metrics}

We define main optimization metrics for our service placement and service placement reconfiguration problems. The placement problem is only considered at $t+1$, just after the placement, while the reconfiguration problem can either be considered at $t+1$ or $t+h$ for the reactive and proactive reconfigurations respectively. To generalize the problem, we consider our metrics at a time $t+h$, where $h \in [1, H]$, with $H$ our prediction horizon. 

\subsection{Overall power consumption}

We compute the CPU usage of each node at time $t+h$ as in~\eqref{eq:cpu_obj} and~\eqref{eq:cpu_perc_obj}, from the prediction of CNF $f$ workload $\widetilde{W}^{CPU}_f$. Then, in~\eqref{eq:power_obj}, we derive the power consumption using the power model of the node.

\begin{align}\label{eq:cpu_obj}
    \widetilde{CPU}_n(t+h) &= \sum_{f \in F} X(n,f,t+h) \cdot \widetilde{W}^{CPU}_f(t+h).
\end{align}

\begin{align}\label{eq:cpu_perc_obj}
    \widetilde{CPU}_n^{\%}(t+h) &= \min\left(1, \frac{\widetilde{CPU}_n(t+h)}{CPU_n^{alloc}}\right).
\end{align}

\begin{align}\label{eq:power_obj}
    \widetilde{P}_n(t+h) &= PM_n\left(\widetilde{CPU}_n^{\%}(t+h)\right).
\end{align}

\subsection{Brown power consumption}

The brown power consumption is then obtained in~\eqref{eq:brown_obj} by subtracting the prediction of power generation $\widetilde{P}^{green}_n$ to the estimated power consumption.

\begin{align}
    \widetilde{P}^{brown}_n(t+h) = &\begin{cases}
        \widetilde{P}_n(t+h) - \widetilde{P}_n^{green}(t+h) & \text{if } \widetilde{P}_n(t+h) > \widetilde{P}_n^{green}(t+h), \\
        0 & \text{otherwise.}
    \end{cases} \label{eq:brown_obj}
\end{align}

\subsection{Wasted power}

Similarly, the wasted power can be obtained using~\eqref{eq:wasted_obj} by subtracting the estimated power consumption to the prediction of power generation $\widetilde{P}^{green}_n$.

\begin{align}
    \widetilde{P}^{wasted}_n(t+h) = &\begin{cases}
        \widetilde{P}_n^{green}(t+h) - \widetilde{P}_n(t+h)  & \text{if } \widetilde{P}_n(t+h) < \widetilde{P}_n^{green}(t+h), \\
        0 & \text{otherwise.}
    \end{cases} \label{eq:wasted_obj}
\end{align}

\subsection{Service green ratio}

From both the prediction of green power generation $\widetilde{P}^{green}_n$ and our estimation of power consumption $\widetilde{P}_n$ in~\eqref{eq:power_obj}, we can compute the \textit{green ratio} of a node $n$ at time $t+h$ as in~\eqref{eq:green_node}, that represents the percentage of the energy consumed on the node that is green at time slot $t+h$. From this ratio, we define the green ratio of a service $s$ as~\eqref{eq:green_svc}, assuming that the green ratio of a CNF is the ratio of the node it is placed on. The satisfaction of the green ratio constraint for a service is then defined in~\eqref{eq:green_valid}, where $G^c_s$ is the minimum required green ratio for service $s$. Since we aim to avoid interruptions in our system, defining an acceptable threshold of green energy usage for our services is an alternative to purely maximize the green usage of our infrastructure. This constraint could be negotiated between the service provider and the infrastructure provider, that orchestrate the wanted services.

\begin{align}\label{eq:green_node}
    \widetilde{G}_n(t+h) = \begin{cases}
        \frac{\widetilde{P}_n^{green}(t+h)}{\widetilde{P}_n(t+h)} & \text{if } \widetilde{P}_n^{green}(t+h) < \widetilde{P}_n(t+h) \\
        1 & \text{otherwise}.
    \end{cases}
\end{align}

\begin{align}\label{eq:green_svc}
    \widetilde{G}_s(t+h) = \frac{1}{|F_s|} \sum_{f \in F_s} \sum_{n \in N} X(n,f,t+h) \cdot \widetilde{G}_n(t+h).
\end{align}

\begin{align}\label{eq:green_valid}
    \widetilde{G}^{valid}_s(t+h) = 
    \begin{cases}
        1 & \text{if } \widetilde{G}_s(t+h) \geq G^c_s, \\
        0 & \text{otherwise}.
    \end{cases}
\end{align}

\subsection{Service concentration ratio}

To minimize the impact of node departures on service continuity, we define the \textit{service concentration ratio (SCR)} of a service $s$ at time $t+h$ as in~\eqref{eq:scr_obj}, where $|F_s|$ is the total number of CNFs of service $s$ and $n_s(t)$ is the highest number of CNFs that on the same node at time $t$. The goal of this metric is to minimize the amount of CNFs on the same node in case of departure. $r \in \{0,1\}$ is a constant to define a threshold that is acceptable, e.g. $0.5$ for a maximum of $50\%$ of the service's CNFs on the same node. The idea behind this metric is to avoid clustering too much CNFs of the same service on the same node, that will cause an interruption of the service in case of departure. This objective is contradictory with the service green ration satisfaction, that would prioritize the choice of the ``best'' node, i.e. the greenest. Note that the SCR metric is different from load balancing, since we still can have large difference in the load of our nodes.

\begin{align}\label{eq:scr_obj}
    SCR_s(t+h) = \begin{cases}
        r \cdot \frac{|F_s|}{n_s(t+h)} & \text{if } n_s(t+h) > r \cdot |F_s| \\
        1 & \text{otherwise.}
    \end{cases}    
\end{align}

\subsection{Migration cost}

Finally, we define the migration cost as in~\eqref{eq:migration_cost} for each migrated CNF $f \in \widehat{F}$, where $t_{init}(f)$ is the initialization time of a CNF $f$. Our logic behind this is that to preserve the continuity of the service, the original instance of the migrated CNF must continue to run until the new instance is initialized, leading to a surplus of consumption during this migration.

\begin{align}\label{eq:migration_cost}
    m_f(t+h) = \sum_{n \in N} M(n,f,t+h) \cdot t_{init}(f) \cdot CPU^{req}_f.
\end{align}

\subsubsection{GREENING}

\begin{algorithm}[!h]
\caption{Greedy Placement}
\label{alg:greedy}
\KwIn{$N$: Nodes, $F_s$: CNFs to place, $X$: Placement Matrix}
\KwOut{Placement mapping at $t+1$}
$N^{(\tau)} \leftarrow sort(N, P^{green}_n - P_n)$\;
\For{$f \in F_s$}{
    \For{$n \in N^{(\tau)}$}{
        \If{$n$ has enough resources for $f$}{
            $X(n,f,t+1) \leftarrow 1$\;
            break\;
        }
    }
    \If{$\sum_{n \in N} X(n,f,t+1) = 0$}{
        $X(0,f,t+1) \leftarrow 1$\;
    }
}
\Return{$X$}\;
\end{algorithm}

\begin{algorithm}[!h]
\caption{GREENING (migration)}
\label{alg:greening}
\KwIn{$N$: Nodes, $F$: Running CNFs, $X$: Placement Matrix, $M$: Migration Matrix}
\KwOut{Migration plan at $t+1$}
$N^{(\tau)} \leftarrow sort(N, P^{green}_n - P_n)$\;
\For{$n \in N^{(\tau)}$}{
    \If{$P^{green}_n(t) > P^{green}_n(t+1)$}{
        $candidates \leftarrow \{ f \in F,\quad X(n, f, t) = 1\}$\;
        \While{$P_n(t+1) > P^{green}_n(t+1)$}{
            \If{$candidates = \emptyset$}{
                break\;
            }
            $sorted\_candidates \leftarrow sort(candidates, t_{init}(f) \cdot CPU^{req}_f)$\;
            \For{$d \in N^{(\tau)}$}{
                \If{$d = n$}{
                    continue\;
                }
                \If{$sorted\_candidates[0]$ can be placed on $d$ and new $P_n(t+1) \leq P_n^{green}(t+1)$}{
                    $X(d,f,t+1) \leftarrow 1$\;
                    $X(n,f,t+1) \leftarrow 0$\;
                    $M(d,f,t+1) \leftarrow 1$\;
                }
            }
            Remove $sorted\_candidates[0]$ from $candidates$\;
        }
    }
}
\Return{$M$}\;

\end{algorithm}

\subsubsection{Lb-diversity}

\begin{algorithm}[!h]
\caption{Lb-diversity}
\label{alg:lb-diversity}
\KwIn{$N$: Nodes, $F_s$: CNFs to place, $X$: Placement Matrix}
\KwOut{Placement mapping at $t+1$}
$\bar{F} \leftarrow$ sort $F_s$ by sum of CPU and RAM ranking\;
\For{$f \in \bar{F}$}{
    $candidates \leftarrow \emptyset$\;
    $m \leftarrow -\infty$\;
    \For{$n \in N$}{
        $n \leftarrow \sum_{k \in F} X(n,k,t)$\;
        \If{$n < m$}{
            $candidates \leftarrow \{ n \}$\;
        }
        \ElseIf{$n = m$}{
            $candidates \leftarrow candidates \cup \{n\}$\;
        }
    }

    $\bar{N} \leftarrow$ sort $candidates$ by sum of CPU and RAM usage\;

    \For{$n \in \bar{N}$}{
        \If{$f$ can be placed on $n$}{
            $X(n,f,t+1) = 1$\;
        }
    }
}
\Return{$X$}\;
\end{algorithm}
